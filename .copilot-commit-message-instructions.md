# Git Commit Message Specs

Analyze the current git status and diffs for the provided files/directories (or the entire project if none are specified) to generate a professional, concise, and structured commit message.

Guidelines:

1. **Format:** Follow the Conventional Commits specification: `<type>(<scope>): <description>`.
   - Types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
   - Scope: Optional (e.g., `core`, `config`, `deps`).
   - Subject: Use the imperative, present tense ("add", "fix"), do not capitalize the first letter, and no trailing period.
2. **Context:** Examine both staged (`git diff --staged`) and unstaged changes for the target files/directories to understand the full intent.
3. **Body (Optional):** If the changes are complex, add a brief bulleted list explaining "what" and "why" (not "how").
4. **Language:** The commit message must be in English.
5. **Output:** Provide ONLY the final commit message text, ready for use. Do not include markdown code blocks or conversational filler.

Process:

- If file or directory arguments are provided, use `git status <path>` and `git diff --staged <path>` to focus the analysis. Otherwise, check the entire repository.
- Determine the most appropriate commit type based on the targeted changes.
- Construct the final message following the strict format.
